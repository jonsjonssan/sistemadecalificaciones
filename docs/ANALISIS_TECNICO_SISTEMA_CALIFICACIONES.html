1. Resumen Ejecutivo

El Sistema de Calificaciones Escolar es una aplicación web full-stack construida sobre Next.js 16 con React 19 y TypeScript, orientada a centros educativos que imparten grados de 2° a 9°. Su propósito es digitalizar el registro de calificaciones trimestrales, asistencia, generación de boletas, reportes estadísticos y alertas tempranas de estudiantes en riesgo académico.

El sistema adopta un modelo multi-escuela (multi-tenant) donde cada Escuela aísla sus usuarios, estudiantes, materias y calificaciones. La autenticación se implementa con sesiones firmadas mediante HMAC-SHA256 (cookie session), no con NextAuth a pesar de que la dependencia esté en package.json. La base de datos es PostgreSQL gestionada a través de Prisma ORM, con despliegue recomendado en Vercel + Neon.

Objetivo del análisis: describir con rigor técnico cómo funciona el sistema, qué tecnologías lo componen, las fórmulas matemáticas que rigen cada cálculo, sus fortalezas, sus puntos débiles y una propuesta priorizada de mejoras que puedan aplicarse de forma secuencial o con impacto exponencial.

3. Arquitectura y Funcionamiento

3.1. Patrón arquitectónico

El sistema sigue una arquitectura monolítica full-stack sobre Next.js. El frontend (Server Components + Client Components) y el backend (API Routes) coexisten en el mismo repositorio y despliegue. La capa de datos se abstrae con Prisma y se accede desde los endpoints REST.

Cliente

  • Server Components para carga inicial.
  • Client Components para interactividad.
  • Zustand para estado global.
  • TanStack Query para cacheo de servidor.
  • Service Worker para PWA.

Servidor

  • API Routes en src/app/api.
  • Sesiones firmadas con HMAC-SHA256.
  • Validación con Zod en cada endpoint.
  • Prisma accede a PostgreSQL.
  • Logs de auditoría e historial de cambios.

3.2. Flujo de autenticación

  1. El usuario envía email, password y escuelaId a POST /api/auth/login.
  2. El servidor busca el usuario por email + escuelaId.
  3. Se compara el hash con bcrypt.compare; si falla, se compara con un hash dummy para evitar timing attacks.
  4. Se crea un registro en LoginSession y se firma una cookie session con HMAC-SHA256.
  5. La cookie contiene: id, email, rol, escuelaId, asignaturasAsignadas, gradosAsignados, etc.
  6. En cada request a /api/*, el middleware aplica rate-limiting por IP y headers de seguridad.
  7. Los endpoints usan verifySession y helpers como requireAdmin para autorizar.

Nota importante Aunque next-auth está en dependencias, no se utiliza. La autenticación es 100% custom. Esto puede generar confusión y deuda técnica.

3.3. Flujo de calificaciones

  1. El docente selecciona grado, materia y trimestre.
  2. El frontend obtiene la configuración de actividades (ConfigActividad) para esa materia y trimestre.
  3. Se renderiza una matriz editable: N actividades cotidianas, M integradoras y opcionalmente el examen.
  4. Al guardar, useGrading envía un POST a /api/calificaciones con los arrays de notas.
  5. El backend valida con Zod, calcula promedios, recalcula el promedio final ponderado y persiste en una transacción Prisma.
  6. Durante la transacción se registran el historial de cambios por celda y un AuditLog resumido.

4.2. Normalización de notas

El sistema mantiene dos representaciones de las notas:

  • Agregada: Calificacion.calificacionAC, calificacionAI, examenTrimestral, promedioFinal.
  • Detalle normalizado: filas en NotaActividad con tipo (cotidiana/integradora/examen) y numeroActividad.

Esto permite reconstruir el historial de cada celda y regenerar promedios cuando cambian los porcentajes de ConfigActividad.

5. Matemática de Calificaciones

5.1. Datos de entrada

Para cada estudiante, materia y trimestre, el sistema requiere:

  • AC = [ac₁, ac₂, ..., acₙ]: notas de actividades cotidianas.
  • AI = [ai₁, ai₂, ..., aiₘ]: notas de actividades integradoras.
  • EX = [ex₁, ex₂, ..., exₖ]: notas de examen (normalmente k=1).
  • recuperacion: nota de recuperación trimestral (opcional).
  • ConfigActividad: porcentajes pAC, pAI, pEX que deben sumar 100%.

5.2. Promedio simple

Promedio(AC) = (Σ acᵢ) / n    donde acᵢ ≠ null
Promedio(AI) = (Σ aiᵢ) / m    donde aiᵢ ≠ null
Promedio(EX) = (Σ exᵢ) / k    donde exᵢ ≠ null

5.3. Promedio final trimestral

PF = Promedio(AC) × (pAC/100) + Promedio(AI) × (pAI/100) + Promedio(EX) × (pEX/100)

Si existe recuperación:

PF_final = min(10, PF + recuperacion)

El valor se almacena sin redondear en la base de datos, pero el frontend suele mostrar Math.round(pf) para determinar estados y colores.

5.4. Promedio anual por materia

El sistema no define un modelo explícito de promedio anual; se calcula sobre la marcha promediando los promedios finales de los trimestres con datos:

PromedioAnual(Materia) = (PF_T1 + PF_T2 + PF_T3) / t    donde t = cantidad de trimestres con datos

5.5. Clasificación de escala

Usando los umbrales configurables umbralCondicionado (default 4.5) y umbralAprobado (default 6.5):

Si PF < umbralCondicionado → REPROBADO
Si umbralCondicionado ≤ PF < umbralAprobado → CONDICIONADO
Si PF ≥ umbralAprobado → APROBADO

5.6. Promedio general del estudiante

PromedioGeneral = Σ PromedioAnual(Materiaᵢ) / M    donde M = cantidad de materias con datos

5.7. Asistencia

El porcentaje de asistencia pondera las tardanzas como medio día:

Asistencia% = ((presentes + tardanzas × 0.5) / total) × 100

5.8. Agente de riesgo (puntaje)

El agente asigna un puntaje de riesgo entre 0 y 1 sumando factores:

FactorCondiciónPeso
Promedio muy bajoPG < umbralCondicionado+0.40
Promedio bajoumbralCondicionado ≤ PG < umbralAprobado+0.25
Tendencia fuertemente negativatendencia < -0.5+0.20
Tendencia negativatendencia < -0.2+0.10
Materias reprobadascada una+0.20 × count
Materias condicionadascada una+0.10 × count
Asistencia críticaasistencia% < 70+0.30
Asistencia bajaasistencia% < 80+0.15
PuntajeRiesgo = min(1, Σ pesos)
tipo = riesgo_alto (≥0.6) | riesgo_medio (≥0.4) | asistencia_critica | bajo_rendimiento

5.9. Tendencia de promedios

Tendencia = Σ(PFᵢ - PFᵢ₋₁) / (t - 1)    para i = 2..t

6. Seguridad y Multi-tenencia

6.1. Aislamiento por escuela

  • Cada modelo incluye escuelaId.
  • La sesión transporta escuelaId; los endpoints filtran por este campo.
  • El email puede repetirse entre escuelas gracias a @@unique([email, escuelaId]).

6.2. Autorización basada en roles

RolPermisos generales
superadminAcceso global; puede gestionar escuelas y usuarios.
admin, admin-directora, admin-codirectoraGestión completa de su escuela: usuarios, grados, estudiantes, materias, configuración.
docenteIngresa calificaciones y asistencia solo de materias y grados asignados.
docente-orientadorSimilar a docente con funciones orientadoras.

6.3. Headers de seguridad

Next.js configura CSP, X-Frame-Options, X-XSS-Protection, COOP, COEP, Referrer-Policy y Permissions-Policy. El middleware aplica rate limiting de 60 req/min por IP.

6.4. Observaciones de seguridad

  • Atención El middleware no realiza autenticación; depende de cada endpoint.
  • Atención Algunos endpoints usan any en la sesión y construyen where dinámicos.
  • Atención El script mensual del agente usa un header Authorization: Bearer que el API no valida.

7. Offline, Resiliencia y PWA

7.1. Service Worker

El archivo public/sw.js implementa estrategias de cacheo:

  • Pages: network-first para la app shell.
  • Static (_next/static): cache-first.
  • Images/fonts: cache-first.
  • API: network-only (no cachea datos dinámicos).

7.2. IndexedDB

src/lib/offline-database.ts usa idb para almacenar:

  • Respuestas cacheadas de GET con TTL.
  • Cola de mutaciones pendientes (offlineQueue).
  • Datos locales de estudiantes, calificaciones y configuraciones.

7.3. fetchWithCache

src/lib/resilient-fetch.ts proporciona:

  • Timeout de 15 segundos.
  • Reintentos con backoff exponencial: delay = 1000 × 2^attempt (máx. 10s).
  • Circuit breaker: se abre tras 5 fallos, medio abierto a los 30s.
  • Encolamiento automático de mutaciones fallidas.

7.4. Sincronización

Cuando el navegador vuelve a estar online, NetworkStatusIndicator llama a syncOfflineQueue() para reintentar las operaciones pendientes.

8. Agente de Alertas Predictivo

8.1. Funcionamiento

El agente analiza periódicamente (o bajo demanda) todos los estudiantes activos de una escuela y genera alertas de riesgo académico. Se compone de:

  • rules.ts: motor de puntuación y recomendaciones.
  • /api/agent/monitor: endpoint para ejecutar y persistir alertas.
  • /api/agent/query: endpoint de consultas en lenguaje natural simple.
  • /api/alerts/predictive: análisis predictivo con tendencias, correlaciones y probabilidades de reprobación.

8.2. Predicción de reprobación

probabilidad = base(promedioGeneral) + 10 × materiasCríticas
probabilidad = min(100, probabilidad)

Donde la base es: 90% si PG < 4.0; 75% si < 4.5; 60% si < 5.0; 30% si < 5.5; 10% en otro caso.

10. Fortalezas (Pros)

  • Stack moderno: Next.js 16, React 19, Tailwind 4, Prisma 6, TypeScript 5.
  • Multi-tenencia real: aislamiento por escuela en todos los modelos.
  • Auditoría completa: historial por celda y logs de acciones.
  • Resiliencia: cola offline, reintentos, circuit breaker y PWA.
  • UX cuidada: modo oscuro, animaciones, responsive, shadcn/ui.
  • Agente predictivo: alertas tempranas con recomendaciones.
  • Flexibilidad curricular: pesos y cantidad de actividades configurables por materia/trimestre.
  • Tests: cobertura unitaria, de lógica y E2E.
  • Reportes: boletas, cuadro de honor, informes técnicos, exportación Excel/PDF.
  • Seguridad básica sólida: CSP, headers, rate limiting, HMAC en cookies.

11. Debilidades y Deuda Técnica

  • NextAuth sin uso: dependencia instalada pero no implementada; confusión arquitectónica.
  • Tipado relajado: uso de any en helpers de sesión y transacciones Prisma.
  • Cálculos duplicados: lógica de promedio final repetida en API y utilidades.
  • Recuperación anual desconectada: no se integra automáticamente en el cálculo del promedio anual.
  • Magic numbers: valores hardcodeados (35/30, umbrales, topes) dispersos en el código.
  • Validación de porcentajes: el cálculo permite desviaciones pequeñas pero no hay un único validador central.
  • Escalabilidad del agente: análisis secuencial N+1 por estudiante; puede ser lento con miles de registros.
  • Script mensual: usa Bearer token no validado por el endpoint.
  • Dependencias pesadas: MDX editor, DnD kits, Zod validation error, etc., algunas quizás poco usadas.
  • Middleware limitado: no autentica; solo rate limiting y headers.
  • Documentación de API: no hay OpenAPI/Swagger ni contratos formales.
  • Internacionalización: next-intl está instalado pero la app parece monolingüe (es-SV).

13. Conclusiones

El Sistema de Calificaciones Escolar es una aplicación madura y funcional para centros educativos pequeños y medianos. Su fortaleza radica en la combinación de una experiencia de usuario moderna, multi-tenencia, auditoría detallada y capacidades offline. Las fórmulas de cálculo son directas pero efectivas, y la configurabilidad de pesos por materia le otorga flexibilidad real.

Sin embargo, acumula deuda técnica significativa: duplicación de lógica de cálculo, tipado relajado, dependencias no usadas y procesos potencialmente escalables de forma secuencial. Las mejoras de mayor retorno se concentran en centralizar la matemática, tipar sesiones y transacciones, resolver la dualidad NextAuth/custom auth y preparar el motor de alertas para escalar.

A corto plazo, el sistema es mantenible y desplegable. A mediano plazo, las mejoras exponenciales (jobs en background, analytics y ML) lo convertirían en una plataforma de inteligencia educativa, no solo un registro de notas.
